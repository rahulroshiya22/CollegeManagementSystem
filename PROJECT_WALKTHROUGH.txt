PROJECT OVERVIEW
================================================================================
A comprehensive microservices-based College Management System built with 
.NET 8, featuring modern architecture patterns including API Gateway, 
Event-Driven Communication, and Docker containerization.

TECHNOLOGY STACK
================================================================================

Backend Technologies:
--------------------
• .NET 8                    - Core framework for all microservices
• ASP.NET Core Web API      - RESTful API development
• Entity Framework Core     - ORM for complex queries and migrations
• Dapper                    - High-performance data access layer
• Ocelot                    - API Gateway for routing and aggregation
• RabbitMQ (CloudAMQP)      - Message broker for async communication
• SQL Server                - Relational database management
• Docker                    - Containerization for deployment
• Swagger/OpenAPI           - API documentation

Frontend/Dashboard:
------------------
• HTML5, CSS3, JavaScript   - Modern web dashboard
• Chart.js                  - Data visualization
• FontAwesome               - Icon library
• Glassmorphism Design      - Modern UI/UX patterns


ARCHITECTURE OVERVIEW
================================================================================

Data Flow:
----------
Client/Browser → API Gateway (Port 7000) → Microservices → SQL Server
                                         → RabbitMQ (Event Bus)

The system follows a microservices architecture where:
1. All client requests go through the Ocelot API Gateway
2. Gateway routes requests to appropriate microservices
3. Services communicate asynchronously via RabbitMQ
4. Each service has its own database (Database-per-Service pattern)


MICROSERVICES BREAKDOWN
================================================================================

1. API Gateway Service
   ------------------
   Port: 7000
   Technology: Ocelot
   Purpose: Central entry point for all client requests
   Features:
   - Request routing to downstream services
   - Load balancing
   - Rate limiting
   - API aggregation
   - CORS configuration

2. Student Service
   ---------------
   Port: 7001
   Database: StudentDB
   Purpose: Manage student profiles and academic records
   
   Key Endpoints:
   • GET    /api/Student              - Get all students (with pagination)
   • GET    /api/Student/{id}         - Get student by ID
   • POST   /api/Student              - Create new student
   • PUT    /api/Student/{id}         - Update student details
   • DELETE /api/Student/{id}         - Delete student
   • PUT    /api/Student/{id}/status  - Update student status
   • GET    /api/Student/{id}/exists  - Check student existence
   
   Features:
   - Email validation and uniqueness check
   - Pagination support
   - Search and filter capabilities
   - Status management (Active/Inactive/Graduated)

3. Course Service
   --------------
   Port: 7002
   Database: CourseDB
   Purpose: Manage course catalog and offerings
   
   Key Endpoints:
   • GET    /api/Course                    - Get all courses
   • GET    /api/Course/{id}               - Get course by ID
   • GET    /api/Course/department/{id}    - Get courses by department
   • GET    /api/Course/semester/{sem}     - Get courses by semester
   • POST   /api/Course                    - Create new course
   • PUT    /api/Course/{id}               - Update course
   • DELETE /api/Course/{id}               - Delete course
   • GET    /api/Course/{id}/exists        - Check course existence
   
   Features:
   - Department-wise course management
   - Semester categorization
   - Credit hour tracking
   - Course prerequisites management

4. Enrollment Service
   ------------------
   Port: 7003
   Database: EnrollmentDB
   Purpose: Handle student course enrollments
   
   Key Endpoints:
   • GET    /api/Enrollment                 - Get all enrollments
   • GET    /api/Enrollment/{id}            - Get enrollment by ID
   • GET    /api/Enrollment/student/{id}    - Get enrollments by student
   • GET    /api/Enrollment/course/{id}     - Get enrollments by course
   • POST   /api/Enrollment                 - Create enrollment
   • DELETE /api/Enrollment/{id}            - Drop enrollment
   
   Features:
   - Student existence validation via Student Service
   - Course existence validation via Course Service
   - Event publishing to RabbitMQ on enrollment creation
   - Status tracking (Enrolled/Dropped/Completed)

5. Fee Service
   -----------
   Port: 7004
   Database: FeeDB
   Purpose: Handle fee calculations and payment processing
   
   Key Endpoints:
   • GET    /api/Fee                   - Get all fees
   • GET    /api/Fee/{id}              - Get fee by ID
   • GET    /api/Fee/student/{id}      - Get fees by student
   • POST   /api/Fee                   - Create fee record
   • PUT    /api/Fee/{id}              - Update fee
   • DELETE /api/Fee/{id}              - Delete fee
   
   Features:
   - Event-driven fee generation (listens to Enrollment events)
   - Payment status tracking (Pending/Paid/Overdue)
   - Due date management
   - Amount calculations

6. Attendance Service
   ------------------
   Port: 7005
   Database: AttendanceDB
   Purpose: Track student attendance
   
   Key Endpoints:
   • GET    /api/Attendance              - Get all attendance records
   • GET    /api/Attendance/{id}         - Get attendance by ID
   • POST   /api/Attendance              - Mark attendance
   • PUT    /api/Attendance/{id}         - Update attendance
   • DELETE /api/Attendance/{id}         - Delete attendance
   
   Additional Features (Dapper-based):
   • POST   /api/AttendanceDapper/bulk   - Bulk insert attendance
   • POST   /api/AttendanceDapper/upload/csv    - CSV file upload
   • POST   /api/AttendanceDapper/upload/excel  - Excel file upload
   • GET    /api/AttendanceDapper/template/csv   - Download CSV template
   • GET    /api/AttendanceDapper/template/excel - Download Excel template
   
   Features:
   - Date-wise attendance tracking
   - Status management (Present/Absent/Late)
   - Bulk operations support
   - File upload capabilities
   - High-performance Dapper implementation


EVENT-DRIVEN COMMUNICATION
================================================================================

Message Flow:
------------
EnrollmentService → RabbitMQ → FeeService

When a student enrolls in a course:
1. Enrollment is created in EnrollmentService
2. EnrollmentCreatedEvent is published to RabbitMQ
3. FeeService listens to the event
4. FeeService automatically generates fee record for the enrollment

RabbitMQ Configuration:
----------------------
Provider: CloudAMQP (cloud-hosted RabbitMQ)
Exchange: enrollment_exchange
Queue: fee_queue
Routing Key: enrollment.created

Benefits:
• Decoupled services
• Asynchronous processing
• Reliability through message persistence
• Scalability


DATABASE DESIGN
================================================================================

Each microservice maintains its own database:

StudentDB:
---------
Students table:
- StudentId (PK)
- FirstName
- LastName
- Email (Unique)
- DateOfBirth
- EnrollmentDate
- Status (Active/Inactive/Graduated)

CourseDB:
--------
Courses table:
- CourseId (PK)
- CourseName
- CourseCode (Unique)
- Credits
- DepartmentId
- Semester

EnrollmentDB:
------------
Enrollments table:
- EnrollmentId (PK)
- StudentId (FK - logical, validated via API)
- CourseId (FK - logical, validated via API)
- EnrollmentDate
- Status (Enrolled/Dropped/Completed)

FeeDB:
-----
Fees table:
- FeeId (PK)
- StudentId (FK - logical)
- EnrollmentId (FK - logical)
- Amount
- DueDate
- Status (Pending/Paid/Overdue)

AttendanceDB:
------------
Attendance table:
- AttendanceId (PK)
- StudentId (FK - logical)
- CourseId (FK - logical)
- Date
- Status (Present/Absent/Late)


GETTING STARTED
================================================================================

Prerequisites:
-------------
• .NET 8 SDK
• SQL Server (LocalDB or full instance)
• Docker Desktop (optional, for containerization)
• RabbitMQ account on CloudAMQP (or local RabbitMQ)
• Visual Studio 2022 or VS Code

Setup Steps:
-----------
1. Clone/Open the project:
   cd "d:\ADV DOT NET ums  project\CollegeManagementSystem"

2. Update Connection Strings:
   Each service has appsettings.json with connection strings
   Update if needed for your SQL Server instance

3. Update RabbitMQ Configuration:
   Edit CloudAMQPBus.cs in CMS.Common.Messaging
   Add your CloudAMQP connection string

4. Start All Services:
   Option 1 - PowerShell:
   powershell -ExecutionPolicy Bypass -File ./StartAllServices.ps1
   
   Option 2 - Manual:
   Start each service individually from their project folders

5. Access the Dashboard:
   Open Dashboard.html in your browser
   Click "Project Info & Architecture" for detailed overview

6. Test APIs:
   Navigate to Swagger documentation:
   - Student Service:     https://localhost:7001/swagger
   - Course Service:      https://localhost:7002/swagger
   - Enrollment Service:  https://localhost:7003/swagger
   - Fee Service:         https://localhost:7004/swagger
   - Attendance Service:  https://localhost:7005/swagger


TESTING THE WORKFLOW
================================================================================

Complete Flow Test:
------------------
1. Create a Student:
   POST https://localhost:7001/api/Student
   {
     "firstName": "John",
     "lastName": "Doe",
     "email": "john.doe@example.com",
     "dateOfBirth": "2000-01-01",
     "enrollmentDate": "2024-01-01"

     "courseName": "Introduction to Programming",
     "courseCode": "CS101",
     "credits": 3,
     "departmentId": 1,
     "semester": 1
   }

   POST https://localhost:7003/api/Enrollment
   {
     "studentId": 1,

   → This automatically triggers fee generation in FeeService!

   GET https://localhost:7004/api/Fee/student/1

5. Mark Attendance:
   POST https://localhost:7005/api/Attendance
   {
     "studentId": 1,
     "courseId": 1,
     "date": "2024-01-20",

   }

DEPLOYMENT
================================================================================

Docker Deployment:
-----------------
Each service includes a Dockerfile for containerization.

1. Build Docker images:
   docker build -t cms-student-service ./CMS.StudentService
   docker build -t cms-course-service ./CMS.CourseService
   (repeat for all services)

2. Run containers:
   docker run -p 7001:80 cms-student-service
   (repeat with appropriate ports)

3. Use Docker Compose (recommended):
   Create docker-compose.yml to orchestrate all services


TROUBLESHOOTING
================================================================================

Common Issues:
-------------
1. Port Conflicts:
   - Ensure ports 7000-7005 are available
   - Check Windows Firewall settings

2. Database Connection Issues:
   - Verify SQL Server is running
   - Check connection strings in appsettings.json
   - Ensure LocalDB is installed if using (localdb)

3. RabbitMQ Connection Issues:
   - Verify CloudAMQP credentials
   - Check network connectivity
   - Ensure correct exchange/queue configuration

4. CORS Errors:
   - Each service has CORS configured for localhost
   - Add your domain if accessing from different origin

5. PowerShell Execution Policy:
   - Use: powershell -ExecutionPolicy Bypass -File ./StartAllServices.ps1
