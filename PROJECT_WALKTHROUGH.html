1. Project Overview

🎯 Purpose: A comprehensive microservices-based College Management System designed to handle student records, course management, enrollments, fee processing, and attendance tracking with modern architecture patterns.

Key Features

  • Microservices Architecture: 6 independent services working together
  • API Gateway: Centralized entry point using Ocelot
  • Event-Driven: Asynchronous communication via RabbitMQ
  • Scalable: Each service can scale independently
  • Modern Dashboard: Real-time data visualization
  • RESTful APIs: Complete Swagger documentation

4. Microservices Breakdown

4.1 API Gateway Service PORT 7000

Technology: Ocelot
Purpose: Central entry point for all client requests

Features:

  • Request routing to downstream services
  • Load balancing
  • API aggregation
  • CORS configuration

4.2 Student Service PORT 7001

Database: StudentDB
Purpose: Manage student profiles and academic records

API Endpoints

GET /api/Student

Get all students with pagination and search

GET /api/Student/{id}

Get student by ID

POST /api/Student

Create new student

PUT /api/Student/{id}

Update student details

PUT /api/Student/{id}/status

Update student status (Active/Inactive/Graduated)

DELETE /api/Student/{id}

Delete student

Sample Request

POST /api/Student
{
  "firstName": "John",
  "lastName": "Doe",
  "email": "john.doe@example.com",
  "dateOfBirth": "2000-01-01",
  "enrollmentDate": "2024-01-01"
}

4.3 Course Service PORT 7002

Database: CourseDB
Purpose: Manage course catalog and offerings

API Endpoints

GET /api/Course

Get all courses

GET /api/Course/{id}

Get course by ID

GET /api/Course/department/{id}

Get courses by department

GET /api/Course/semester/{sem}

Get courses by semester

POST /api/Course

Create new course

PUT /api/Course/{id}

Update course

DELETE /api/Course/{id}

Delete course

4.4 Enrollment Service PORT 7003

Database: EnrollmentDB
Purpose: Handle student course enrollments
⚠️ Important: This service validates student and course existence before enrollment and publishes events to RabbitMQ!

API Endpoints

GET /api/Enrollment

Get all enrollments

GET /api/Enrollment/student/{id}

Get enrollments by student

GET /api/Enrollment/course/{id}

Get enrollments by course

POST /api/Enrollment

Create enrollment (triggers fee generation!)

DELETE /api/Enrollment/{id}

Drop enrollment

4.5 Fee Service PORT 7004

Database: FeeDB
Purpose: Handle fee calculations and payment processing
✅ Event-Driven Feature: Automatically listens to enrollment events from RabbitMQ and generates fee records!

API Endpoints

GET /api/Fee

Get all fees

GET /api/Fee/student/{id}

Get fees by student

POST /api/Fee

Create fee record

PUT /api/Fee/{id}

Update fee (e.g., mark as paid)

4.6 Attendance Service PORT 7005

Database: AttendanceDB
Purpose: Track student attendance with bulk upload support
✅ Special Feature: Supports bulk attendance upload via CSV/Excel files using high-performance SqlBulkCopy!

Standard API Endpoints

GET /api/Attendance

Get all attendance records

POST /api/Attendance

Mark attendance

PUT /api/Attendance/{id}

Update attendance

Bulk Operation Endpoints (Dapper)

POST /api/AttendanceDapper/bulk

Bulk insert attendance records

POST /api/AttendanceDapper/upload/csv

Upload CSV file for bulk import

POST /api/AttendanceDapper/upload/excel

Upload Excel file for bulk import

GET /api/AttendanceDapper/template/csv

Download CSV template

GET /api/AttendanceDapper/template/excel

Download Excel template

6. Event-Driven Communication

Message Flow Diagram

📝
Enrollment Service
Publisher
→
📨
RabbitMQ
Message Broker
→
💰
Fee Service
Subscriber

How It Works

Step Action Service
1 Student enrolls in course Enrollment Service
2 Creates enrollment record in database Enrollment Service
3 Publishes EnrollmentCreatedEvent to RabbitMQ Enrollment Service
4 Receives event from queue Fee Service
5 Automatically generates fee record Fee Service
📨 RabbitMQ Configuration:
Provider: CloudAMQP (cloud-hosted)
Exchange: enrollment_exchange
Queue: fee_queue
Routing Key: enrollment.created

5. Database Design

🗄️ Pattern: Database-per-Service - Each microservice maintains its own database for independence and scalability.

Students Table (StudentDB)

Column Type Notes
StudentId INT (PK) Auto-increment
FirstName VARCHAR(100) Required
LastName VARCHAR(100) Required
Email VARCHAR(255) Unique, Required
DateOfBirth DATE -
EnrollmentDate DATE -
Status VARCHAR(50) Active/Inactive/Graduated

Courses Table (CourseDB)

Column Type Notes
CourseId INT (PK) Auto-increment
CourseName VARCHAR(200) Required
CourseCode VARCHAR(50) Unique, Required
Credits INT -
DepartmentId INT -
Semester INT -

Enrollments Table (EnrollmentDB)

Column Type Notes
EnrollmentId INT (PK) Auto-increment
StudentId INT Validated via API call
CourseId INT Validated via API call
EnrollmentDate DATE -
Status VARCHAR(50) Enrolled/Dropped/Completed

7. Getting Started

Prerequisites

  • .NET 8 SDK
  • SQL Server (LocalDB or full instance)
  • Visual Studio 2022 or VS Code
  • RabbitMQ account on CloudAMQP (or local)

Installation Steps

Step 1: Update Connection Strings

Each service has appsettings.json - update SQL Server connection strings if needed.

Step 2: Configure RabbitMQ

Edit CloudAMQPBus.cs in CMS.Common.Messaging with your CloudAMQP connection string.

Step 3: Start All Services

powershell -ExecutionPolicy Bypass -File ./StartAllServices.ps1

Step 4: Open Dashboard

Open Dashboard.html in your browser

Step 5: Access Swagger Documentation

  • Student: https://localhost:7001/swagger
  • Course: https://localhost:7002/swagger
  • Enrollment: https://localhost:7003/swagger
  • Fee: https://localhost:7004/swagger
  • Attendance: https://localhost:7005/swagger

  "lastName": "Doe",

{

  "departmentId": 1,

Step 3: Enroll Student (Triggers Fee!)

  "courseId": 1,

Step 5: Mark Attendance

{
  "studentId": 1,
  "courseId": 1,
  "date": "2024-01-20",
  "status": "Present"
}

10. Deployment with Docker

Docker Build Commands

# Build Docker images for each service
docker build -t cms-student-service ./CMS.StudentService
docker build -t cms-course-service ./CMS.CourseService
docker build -t cms-enrollment-service ./CMS.EnrollmentService
docker build -t cms-fee-service ./CMS.FeeService
docker build -t cms-attendance-service ./CMS.AttendanceService
docker build -t cms-gateway ./CMS.APIGateway

Run Containers

docker run -p 7001:80 cms-student-service
docker run -p 7002:80 cms-course-service
docker run -p 7003:80 cms-enrollment-service
docker run -p 7004:80 cms-fee-service
docker run -p 7005:80 cms-attendance-service
docker run -p 7000:80 cms-gateway
